# Autonomous Crawl Agent - 工程评估报告

> 评估日期：2026-05-05
> 评估人：高级工程师视角
> 项目阶段：早期 MVP（第 2 天，5 月 4 日启动）
> 版本：v2（补充蓝图根本性缺陷分析与视觉能力设计）

---

## 一、项目定位与目标

**最终目标**：开发一个"非常厉害的爬虫 Agent"——能够理解自然语言指令，自主分析任意网站结构，自动选择最优采集策略，执行采集并返回高质量结构化数据。

**当前实际状态**：一个架构正确的确定性爬虫 pipeline，能够对特定模式的页面（百度热搜榜）完成端到端采集，但距离"自主 Agent"还有本质差距。

**核心差距的本质**：当前系统在"解析 HTML"，而不是"理解网页"。整个信息通道只有一条——HTML 文本 → 字符串解析 → CSS 选择器匹配。这个通道是极度窄化的，它丢弃了人类理解网页时最核心的信息来源：视觉。

## 四、核心问题诊断

### 问题 1：没有视觉理解能力（严重度：致命）

**这是当前蓝图最根本的缺失。**

当前系统的信息通道只有一条：HTML 文本 → 字符串解析 → CSS 选择器匹配。人类看网页的方式完全不同——人眼扫一眼就知道"这是商品列表页"、"价格在这里"、"点这个按钮翻页"。当前系统完全不具备这种能力。

**为什么这是致命问题**：

1. **class 名不可靠**：大量网站使用 hash class（`class="sc-dkPtRN kFwGMH"`），纯靠 class 名推断失效
2. **改版即报废**：网站改版后 class 名全变了，所有选择器全部失效
3. **无法区分主次**：无法区分"商品卡片"和"广告/导航/页脚"——它们在 HTML 中都是 div
4. **无法理解布局**：不知道"上方是导航、中间是商品、底部是分页"这种空间关系
5. **SPA 空壳问题**：拿到的 HTML 是空壳，连 class 名都没有，更无法推断

**系统应该有的视觉能力**：

```
输入：一张网页截图
输出：
  - 页面类型：商品列表页 / 详情页 / 首页 / 搜索结果页
  - 主内容区域：坐标 [x1, y1, x2, y2]
  - 可交互元素：分页按钮位置、筛选器位置、排序按钮位置
  - 数据卡片：每个商品卡片的边界框（bounding box）
  - 字段识别：这个位置是标题、那个位置是价格、这个是图片
  - 翻页机制：是 URL 参数翻页、按钮点击、还是无限滚动
```

**技术实现路径**：

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│  截图模块    │────→│  视觉理解模块  │────→│  结构映射模块 │
│ Playwright  │     │ 多模态 LLM   │     │ 视觉→DOM    │
│ 截取全页截图  │     │ 分析页面布局   │     │ 坐标映射     │
└─────────────┘     └──────────────┘     └─────────────┘
                           │
                           ▼
                    ┌──────────────┐
                    │  结构化输出   │
                    │ 页面类型      │
                    │ 内容区域坐标   │
                    │ 字段类型+位置  │
                    │ 可交互元素位置 │
                    └──────────────┘
```

核心思路：**截图 → 多模态 LLM（GPT-4o / Claude Vision）分析 → 结构化理解 → 驱动后续采集**

这样做的好处：
- 不依赖 class/id 名，网站怎么改版都不怕
- 和人看网页的方式一致，泛化能力强
- 可以处理任意网站，不需要预配置
- 视觉信息比 HTML 信息更稳定（改版后视觉布局变化通常小于 DOM 结构变化）

**这不是"nice to have"，这是让 Agent 能处理任意网站的前提条件。** 没有视觉能力，系统永远只能处理"预先见过的网站模式"。

### 问题 2：没有"心智模型"（Mental Model）（严重度：高）

当前系统对每个页面是独立处理的，没有对网站的整体理解。

**现状**：拿到一个 URL → 分析这个页面的 HTML → 提取数据。结束。

**缺失的能力**：
- 不知道这个网站有几级分类
- 不知道列表页和详情页的导航关系
- 不知道翻页机制是什么（URL 参数？按钮点击？无限滚动？）
- 不知道哪些页面需要登录才能访问
- 不知道数据分布在哪些页面（有的价格在列表页就有，有的要点进详情页）

**系统应该构建的站点心智模型**：

```
探索阶段（5-10 秒）：
  1. 打开首页截图 → 识别导航结构 → 发现分类入口
  2. 点击一个分类 → 截图 → 识别这是列表页
  3. 观察列表页 → 识别翻页方式 → 识别卡片结构
  4. 点击一个商品 → 截图 → 识别这是详情页
  5. 对比列表页和详情页 → 建立"哪些数据在列表页有，哪些要去详情页"

输出：站点心智模型
  - 站点类型：电商 / 内容 / 社交 / 工具
  - 导航结构：首页 → 分类列表 → 商品详情
  - 翻页方式：URL 参数 page=1,2,3...
  - 数据分布：标题/价格在列表页，描述/规格在详情页
  - 反爬等级：低（无保护）/ 中（基础检测）/ 高（Cloudflare）
  - 估计数据量：约 N 个商品，M 页
```

当前 Recon 做的是"分析一个页面的 HTML"，而不是"理解一个网站是怎么运作的"。这两者有本质区别。

---

### 问题 3：没有"探索-利用"策略（严重度：高）

当前系统拿到 URL 后直接开始采集，没有探索阶段。

**问题**：
- 不知道网站有多大（10 个商品还是 10 万个）
- 不知道哪些页面有价值，哪些是空的
- 不知道最优的遍历路径
- 一上来就猛采，可能触发反爬

**应该有的两阶段策略**：

```
探索阶段：
  - 随机采样 5-10 个页面
  - 估算总数据量
  - 识别分页深度
  - 测试反爬敏感度（请求频率阈值）
  - 识别数据结构一致性（所有商品页面结构是否相同）

利用阶段：
  - 根据探索结果制定采集计划
  - 优先采高价值页面
  - 动态调整请求频率
  - 发现异常时回退到探索阶段
```

### 问题 4：这不是 Agent，是 Pipeline（严重度：高）

当前系统的行为是完全确定性的：给定相同的输入，产出完全相同的结果。它没有推理能力、没有决策能力、没有学习能力。

**证据**：
- Planner 用 15 个 if-keyword-in-string 做路由
- Strategy 用 4 层 if-else 做策略选择
- 没有任何节点调用 LLM
- 没有任何节点会根据上下文做非确定性判断

**影响**：
- 遇到训练数据之外的网站模式必然失败
- 无法处理歧义指令（"把好看的都抓下来"）
- 无法从失败中学习（重试只是重跑同一个策略）

---

### 问题 5：没有"容错和自愈"能力（严重度：高）

选择器失效 → 重试 3 次 → 失败。结束。

**问题**：
- 网站改版是常态，不是异常
- 页面 A 采集成功不代表页面 B 也能成功
- 重试同一个策略 3 次没有意义——如果第一次不行，第三次大概率也不行

**应该有的自愈能力**：

```
选择器失效时：
  1. 截图当前页面
  2. 用视觉模型重新识别页面结构
  3. 生成新的选择器
  4. 验证新选择器是否有效
  5. 如果有效，更新站点配置

采集异常时：
  1. 检测是临时错误（网络超时）还是结构性错误（页面变了）
  2. 临时错误 → 等待 + 重试
  3. 结构性错误 → 重新探索 → 更新策略 → 继续
```

当前系统没有区分"临时失败"和"结构性失败"，所有失败都走同一条重试路径。

---

### 问题 6：浏览器模式缺失（严重度：高）

现代网站超过 60% 是 SPA（React/Vue/Next.js），服务端返回的 HTML 是空壳，必须用浏览器渲染才能拿到真实 DOM。当前 Executor 对这类页面完全无能为力。

**证据**：
- `executor.py` 中 `mode == "browser"` 分支没有实际代码
- Playwright 在 requirements 中但从未 import
- Recon 能检测 `rendering == "spa"` 但 Executor 不响应

**影响**：
- 无法采集淘宝、京东、Amazon 等主流电商
- 无法采集大多数现代内容网站
- 项目只能对少数 SSR/静态网站生效

---

### 问题 7：没有"上下文感知"的提取能力（严重度：中）

当前 Extractor 用 CSS 选择器从容器元素中提取文本，没有语义理解。

**问题**：
- 无法处理嵌套结构（商品有多个 SKU，每个 SKU 有不同价格）
- 无法处理条件渲染（有的商品有折扣价，有的没有）
- 无法处理动态内容（价格是 JS 计算的，HTML 里没有）
- 无法理解数据含义（提取到 "¥199.00" 但不知道这是原价还是折扣价）

**应该有的能力**：
- 视觉识别字段类型（不依赖 class 名）
- 理解字段间的语义关系（"原价 ¥299 → 折扣价 ¥199 → 省 ¥100"）
- 处理同一字段的多种格式（"¥199" / "199.00元" / "CNY 199"）
- 识别并处理"没有这个字段"的情况（不是提取失败，是商品本身没有）

---

### 问题 8：单页限制（严重度：中）

整个 pipeline 只处理一个 URL 的一个页面。真实爬虫场景需要：
- 列表页翻页（第 1 页 → 第 2 页 → ...）
- 列表页 → 详情页的跳转
- 多级分类页的遍历

当前架构不支持 URL 队列、页面间导航、数据跨页合并。

---

### 问题 9：没有反爬对抗能力（严重度：中）

当前的"反爬检测"只是检测 HTML 中是否包含 "cloudflare" 等关键词。检测到之后呢？没有任何应对策略。

**缺少**：
- 代理池管理
- 请求频率控制
- Cookie/Session 管理
- 浏览器指纹伪装
- 验证码处理（至少是人工介入接口）

### 问题 10：没有"记忆和学习"能力（严重度：中）

每次采集都是从零开始。采过的网站、成功的策略、踩过的坑，全部丢失。

**问题**：
- 第一次采淘宝花 10 分钟探索，第二次还是花 10 分钟
- 发现了一个网站的有效选择器，下次还得重新推断
- 被某个网站封过 IP，下次还会在同一个坑里摔倒

**应该有的记忆能力**：

```
站点记忆库：
  - 采集过的网站 → 保存站点心智模型
  - 成功的策略 → 保存为可复用模板
  - 失败的教训 → 保存为避坑规则
  - 选择器版本 → 保存历史版本，支持回滚

知识迁移：
  - "这个网站和之前采过的 X 网站结构很像" → 复用 X 的策略
  - "这类电商网站的翻页通常是 page= 参数" → 直接尝试
```

### 问题 11：死代码造成认知混乱（严重度：低）

`tools/recon_tools.py` 包含 4 个用 `@tool` 装饰的函数，全部返回硬编码空值。真正干活的 recon 逻辑在 `tools/html_recon.py`。这两套代码并存，新人无法判断哪个是真的。

**证据**：
- `recon_tools.py` 的 `detect_framework()` 返回 `'{"framework": "unknown", "confidence": 0.0}'`
- `html_recon.py` 的 `detect_framework()` 能正确检测 React/Vue/Next.js
- `recon.py`（Agent）只 import 了 `html_recon`，从未 import `recon_tools`

## 六、推荐路线图（修订版）

> 基于"非常厉害的爬虫 Agent"目标重新规划，视觉能力作为核心能力建设。

### Phase 1：清理 + 基础设施（1 周）

目标：消除技术债，为后续能力建设打好基础。

**任务**：
1. **删除 `recon_tools.py`** —— 消除死代码，统一到 `html_recon.py`
2. **状态模型改 TypedDict** —— 去掉 `@preserve_state`，用 LangGraph 原生 reducer
3. **节点错误边界** —— 每个节点 try-except，异常时优雅终止而非崩溃
4. **加入错误路径测试** —— HTTP 超时、空响应、403/429、畸形 HTML
5. **Executor 加入代理和频率控制** —— 基础的反爬应对能力

### Phase 2：视觉理解能力（2-3 周）★ 核心阶段

目标：让 Agent 能"看懂"网页，从"解析 HTML"跨越到"理解网页"。

**任务**：
1. **Playwright 截图模块** —— 全页截图 + 可视区域截图 + 元素级截图
2. **视觉理解模块** —— 多模态 LLM（GPT-4o/Claude Vision）分析截图
3. **页面类型识别** —— 输入截图，输出：列表页/详情页/首页/搜索页/登录页
4. **内容区域定位** —— 输出主内容区域的 bounding box
5. **字段视觉识别** —— 不依赖 class 名，通过视觉识别标题/价格/图片/评分位置
6. **翻页机制识别** —— 视觉识别分页控件类型和位置
7. **视觉→DOM 坐标映射** —— 视觉坐标映射到具体 DOM 元素，生成 CSS 选择器

**预期成果**：
- 输入任意网站 URL → 截图 → 视觉分析 → 输出页面结构化理解
- 不依赖 class/id 名，网站改版后仍能工作

## 七、架构改进建议

### 7.1 状态模型：从 dict 到 TypedDict

当前图使用 `StateGraph(dict)`，没有类型约束。建议改为：

```python
from typing import TypedDict, Annotated
from operator import add

class CrawlState(TypedDict):
    task_id: str
    user_goal: str
    target_url: str
    recon_report: dict
    crawl_strategy: dict
    extracted_data: dict
    validation_result: dict
    retries: int
    status: str
    messages: Annotated[list[str], add]  # 自动累加
    error_log: Annotated[list[str], add]  # 自动累加
```

这样可以去掉 `@preserve_state` 装饰器，利用 LangGraph 的原生 reducer 机制。

### 7.2 节点错误边界

当前任何节点抛异常都会导致整个图崩溃。建议每个节点用 try-except 包裹，异常时设置 `status="node_failed"` 并记录错误，让图能优雅终止。

### 7.3 Recon 分层

当前 Recon 做了太多事情（框架检测 + 反爬检测 + API 发现 + DOM 推断）。建议拆分为：

- **Recon: 快速扫描** —— 只做框架检测和反爬检测（毫秒级）
- **Recon: 深度分析** —— DOM 推断和选择器生成（需要解析完整 HTML）
- **Recon: 浏览器探查** —— 需要渲染页面才能获取的信息（JS 执行后的 DOM、XHR 请求）

### 7.4 策略模板库

当前策略是每次实时生成的。建议引入策略模板：

```python
STRATEGY_TEMPLATES = {
    "ecommerce_list": {
        "mode": "http",
        "extraction_method": "dom_parse",
        "pagination": {"type": "url_param", "param": "page"},
    },
    "spa_product_detail": {
        "mode": "browser",
        "extraction_method": "browser_render",
        "wait_for": ".product-detail",
    },
}
```

Strategy Agent 的职责变成：选择模板 + 填充具体选择器。

## 八、总结

### 做得好的

1. 架构设计正确，6 节点职责划分清晰
2. LangGraph 工作流骨架完整，重试逻辑正确
3. Recon 的 DOM 推断算法有真实工程价值
4. 存储层设计规范，可直接用于生产
5. 开发日志完整，工程习惯好
6. 百度热搜实测跑通，证明端到端链路可行

### 必须正视的

1. **信息通道极度窄化** —— 只有 HTML 文本一条通道，没有视觉理解能力，这是最根本的限制
2. **没有心智模型** —— 只分析单个页面，不理解网站是怎么运作的
3. **这不是 Agent，是 Pipeline** —— 没有 LLM，没有推理，没有学习
4. **只能采静态 HTML** —— 60%+ 的现代网站无法处理
5. **没有容错自愈** —— 选择器失效 = 任务失败，没有自动修复能力
6. **蓝图完成度约 35%** —— 离"MVP"还有实质差距

### 与目标的本质差距

```
目标：一个"非常厉害的爬虫 Agent"
当前：一个"能采百度热搜的确定性 Pipeline"

差距：
  - "非常厉害" → 需要视觉理解 + 心智模型 + 探索能力 + 自愈能力
  - "Agent"    → 需要 LLM 推理 + 决策能力 + 学习能力
  - "爬虫"     → 需要浏览器模式 + 反爬对抗 + 多页面编排
```

### 最关键的一步

**从"解析 HTML"跨越到"理解网页"——视觉能力是这个跨越的核心。**

当前系统通过 HTML 标签名和 class 名来"猜"页面结构，这本质上是在用间接信号推断。人类不这么看网页——人用眼睛看。截图 → 多模态 LLM 视觉理解 → 结构化输出，这条路径能让 Agent 像人一样理解任意网页，不再依赖 class/id 名等脆弱的间接信号。

接入 LLM 是从"脚本"到"Agent"的跨越；
接入视觉能力是从"只能处理已知模式"到"能处理任意网站"的跨越。

两者缺一不可，但视觉能力是更底层的基础——没有它，LLM 只能基于 HTML 文本推理，信息量严重不足。
