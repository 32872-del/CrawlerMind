# Handoff: Durable Batch Registry and Backpressure

Date: 2026-05-18
Worker: LLM-2026-002

## Registry Design

**SQLite-backed `BatchRegistry`** replaces the in-memory `_jobs` dict in `api/app.py`.

- File: `autonomous_crawler/storage/batch_registry.py`
- Table: `batch_jobs` with columns: `task_id` (PK), `kind`, `status`, `created_at`, `updated_at`, `completed_at`, `job_json`
- Index on `(status, created_at)` for active job queries and cleanup
- WAL mode, busy_timeout=5000, thread-safe with `threading.Lock()`
- Same constructor/connection/initialize pattern as URLFrontier, ProductStore, CheckpointStore

**Methods** (same API shape as the old in-memory functions):
- `register(task_id, kind, job_data)` — unconditional insert
- `try_register(task_id, kind, job_data, max_active)` — capacity-gated insert
- `update(task_id, **kwargs)` — merge into JSON payload
- `get(task_id)` — return job dict or None
- `remove(task_id)` — delete
- `mark_status(task_id, status)` — set status + completed_at for terminal states
- `count_active()` — count running jobs
- `cleanup_stale(retention_seconds)` — remove old terminal jobs
- `list_jobs(status, kind, limit)` — filtered list
- `recover_running()` — find jobs stuck in "running" from prior crash

## Restart Behavior

On FastAPI startup, `_registry.recover_running()` finds all jobs with `status='running'` from a prior session. These are marked as `failed` with `error_code="CRASH_RECOVERY"` and `error="recovered from prior crash"`. This ensures no jobs are silently left in a phantom running state.

## Backpressure Schema

**`BackpressureMonitor`** in `autonomous_crawler/runners/backpressure.py`:

```
BackpressureSignals:
  avg_latency_ms: float
  p95_latency_ms: float
  retry_rate: float          # retried / claimed
  failure_rate: float        # failed / claimed
  quality_loss_rate: float   # checkpoint_errors / claimed
  consecutive_slow_batches: int
  total_batches: int
  recommendation: str        # proceed | slow_down | pause | abort
  reason: str
```

**Recommendation logic** (evaluated in priority order):
1. `failure_rate >= 0.40` → "abort"
2. `avg_latency_ms >= 15000ms` or `consecutive_slow_batches >= 3` → "pause"
3. `failure_rate >= 0.15` or `retry_rate >= 0.25` → "slow_down"
4. `avg_latency_ms >= 5000ms` or `quality_loss_rate >= 0.10` → "slow_down"
5. Otherwise → "proceed"

**Integration**:
- `BatchRunner` accepts optional `backpressure: BackpressureMonitor | None`
- Times each batch, records metrics, checks recommendation after each batch
- On "abort": sets `summary.status = "aborted"`, breaks
- On "pause": sets `summary.status = "paused"`, breaks
- `ProfileLongRunExecutor` creates a `BackpressureMonitor`, passes to `BatchRunner`, includes signals in `ProfileLongRunResult.backpressure`

## Performance / Durability Limits

- Registry DB: single SQLite file at `autonomous_crawler/storage/runtime/batch_registry.sqlite3`
- WAL mode allows concurrent reads; writes serialized via lock
- `job_json` column stores full job data as JSON — flexible but not queryable by individual fields
- `cleanup_stale()` removes terminal jobs older than retention TTL (default 3600s)
- No automatic migration from the old in-memory state — jobs from prior sessions are lost on first deploy (this is expected since the old registry was non-durable)

## API Changes

- `_jobs` dict and `_jobs_lock` removed from `api/app.py`
- All `_register_job`, `_try_register_job`, `_update_job`, `_get_job`, `_remove_job`, `_count_active_jobs`, `_cleanup_stale_jobs` delegate to `_registry`
- `_update_job` now calls `mark_status` when `status` kwarg is present (to update the SQL column)
- New endpoint: `GET /jobs?status=&kind=&limit=` — list jobs with filters
- New startup event: recovers stale running jobs
- `_clear_jobs()` helper exposed for test teardown
