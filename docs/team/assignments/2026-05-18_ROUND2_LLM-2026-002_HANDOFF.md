## Diagnostics Schema

Added `classify_bottlenecks()` and `recommendation_text()` to `backpressure.py`.

**Bottleneck classification** returns a list of dicts, each with:
```
{
  "key": "access_blocking" | "selector_loss" | "pagination_gap" | "transport_pressure" | "retry_pressure" | "quality_loss",
  "severity": "info" | "warn" | "critical",
  "summary": "English one-liner",
  "summary_zh": "Chinese one-liner for frontend display",
  "evidence": { ... }
}
```

Detection logic:
- **access_blocking**: 403/429/captcha/blocked/forbidden/cloudflare keywords in errors + high failure rate
- **selector_loss**: "selector"/"no records"/"empty"/"parse error" keywords in errors
- **pagination_gap**: frontier exhausted early with < 50% success rate
- **transport_pressure**: avg latency >= 10000ms (critical at >= 15000ms)
- **retry_pressure**: retry rate >= 20% (critical at >= 40%)
- **quality_loss**: checkpoint error rate >= 5% (critical at >= 15%)

**Recommendation text** produces:
```
{
  "code": "proceed" | "slow_down" | "pause" | "abort",
  "message_en": "English summary",
  "message_zh": "Chinese summary"
}
```

Context-aware messages vary by bottleneck type (e.g. abort with access_blocking → "任务已中止：检测到访问被封锁").

## Job Operation Endpoints

| Endpoint | Method | Behavior |
|---|---|---|
| `GET /jobs` | GET | List jobs with filters |
| `GET /jobs/{task_id}` | GET | Full job detail |
| `POST /jobs/{task_id}/cancel` | POST | Cancel running/paused job (409 if not cancellable) |
| `POST /profile-runs/{task_id}/cancel` | POST | Cancel running profile run |
| `POST /profile-runs/{task_id}/pause` | POST | Set `pause_requested=True` flag (best-effort) |
| `POST /profile-runs/{task_id}/resume` | POST | Resume paused run (409 if not paused) |

## Pause/Resume/Cancel State

- **Cancel**: Immediately sets status to "cancelled" in registry. Works for both generic jobs and profile runs. Returns 409 if job is already terminal (completed/failed/cancelled).
- **Pause**: Sets `pause_requested=True` flag in job data. This is best-effort — the runner checks this flag between batches. Actual pause depends on the runner reading the flag. Returns 409 if not running.
- **Resume**: Sets status back to "running" and clears pause flag. Only works if current status is "paused". Returns 409 otherwise.

Note: True pause/resume with runner integration is not yet wired — the endpoints set flags and status but the BatchRunner doesn't read `pause_requested` from the registry between batches. This is documented behavior.

## Remaining Long-Run Observability Gaps

1. **Runner-level pause_requested check**: BatchRunner does not poll the registry between batches to check `pause_requested`. True pause requires either a shared flag or registry polling.
2. **Real-time progress streaming**: No WebSocket/SSE endpoint for live progress updates during long runs.
3. **Per-domain throttle state**: BackpressureMonitor is per-run, not per-domain. Multi-site runs don't share throttle signals.
4. **Historical diagnostics**: Diagnostics are computed at run end. No mid-run checkpointing of diagnostics snapshots for post-mortem analysis.
5. **Error clustering**: Error keyword matching is simple substring search. Could benefit from fuzzy matching or error code taxonomy.
