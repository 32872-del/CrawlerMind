# CLM 前端开发规格

日期: 2026-05-18

## 结论

前端第一版本建议做成**网页管理控制台**，不是桌面软件，也不是纯程序入口。

原因很直接:

1. 当前后端已经是 HTTP API 形态，网页最容易接。
2. 多个员工可以并行开发，路径最清晰。
3. 以后如果要打包成桌面软件，可以再加 Electron/Tauri 外壳，不用推翻重来。
4. 面向企业和开发者的工作台，本来就更适合浏览器式控制台。

所以本项目的前端定位是:

```text
Web 工作台 -> 后台任务监控 -> 结果查看/导出 -> 后续再考虑桌面壳
```

不做营销页，不做介绍页，第一屏就进入工作流。

## 前端目标

把 CLM 的使用流程压缩成一条清晰路径:

```text
配置 LLM / 采集参数 -> 输入网址 -> 解析目录与字段 -> 选择采集内容
-> 试跑 100 条 -> 全量运行 -> 查看进度 -> 导出结果
```

前端要解决的不是“展示”，而是“让采集能稳定落地、可配置、可回溯、可导出”。

## 现有后端基础

前端现在已经可以直接对接这些后端能力:

- `POST /catalog/import`
- `POST /site/analyze`

## 前端页面结构

### 1. 首页 / 工作台

必须显示:

- 当前配置摘要
- 最近任务
- 当前运行中的任务
- 快捷入口: 新建任务、导入目录、查看历史、导出结果

### 2. 新建采集任务向导

建议做成分步向导，不要把所有控件堆在一屏。

步骤如下:

1. 基础配置
2. 网站分析
3. 目录导入/编辑
4. 字段选择
5. 运行模式选择
6. 结果导出

### 3. 任务详情页

必须显示:

- 任务状态
- 已采集数量
- 成功率
- 当前步骤
- 错误信息
- 事件流
- 导出入口

### 4. 历史任务页

必须支持:

- 列表查询
- 按状态筛选
- 按时间筛选
- 查看详情
- 重新运行
- 导出历史结果

### 5. 设置页

用于保存全局配置:

- LLM `base_url`
- `api_key`
- `model`
- 并行数量
- 超时和重试
- 浏览器/代理选项
- 默认导出目录

## 核心业务流程

### 流程 A: 新建任务

1. 用户填写 LLM 配置。
2. 用户输入目标站点 URL。
3. 用户导入站点目录，或让系统自动分析。
4. 用户勾选采集字段。
5. 用户选择试跑或全量运行。
6. 用户选择导出格式。
7. 用户提交任务。

### 流程 B: 任务监控

1. 前端轮询任务状态。
2. 页面展示进度、记录数、失败原因。
3. 如有错误，展示最后一条关键事件。
4. 任务完成后显示导出按钮。

### 流程 C: 结果导出

1. 用户选择导出格式。
2. 用户选择输出文件路径。
3. 用户可选字段映射。
4. 点击导出后返回文件路径或下载链接。

## 页面设计要求

- 以工作台为主，不做大横幅、不做宣传页。
- 第一屏必须是可操作界面。
- 表单和表格优先，装饰元素最少。
- 信息密度要高，但要清晰。
- 重要状态用明显标签显示: running / paused / failed / completed。

## 技术建议

### 推荐方案

- `React + Vite + TypeScript`
- UI 组件库: `Ant Design` 或同级别企业型组件库
- 请求层: 统一封装 `fetch/axios`
- 状态管理: 轻量方案即可，先不要过度设计

### 为什么这样选

- 容易招人接手
- 适合企业工具风格
- 与 FastAPI 后端对接简单
- 后续可直接升级为桌面壳

## 前端模块拆分

建议目录:

```text
src/
  api/
  components/
  pages/
  store/
  types/
  utils/
  hooks/
```

模块建议:

- `api/`: 后端接口封装
- `pages/`: 页面
- `components/`: 可复用组件
- `store/`: 全局状态
- `types/`: 请求和响应类型
- `utils/`: 格式化、校验、导出辅助

## 数据模型重点

前端至少要能表达这些数据:

- LLM 配置
- 目标站点配置
- 目录树
- 字段候选项
- 任务提交参数
- 任务状态
- 运行事件
- 导出配置

## 验收标准

前端第一版验收至少满足:

1. 能完成一次完整的采集任务创建。
2. 能导入目录 JSON。
3. 能从站点分析结果中选择字段。
4. 能发起试跑和全量运行。
5. 能查看任务状态和事件。
6. 能导出结果文件。
7. 页面在 Windows 上可直接打开使用。

## 当前后端缺口

前端可以先做，但有三件后端能力后面要补:

1. 更强的目录发现。
2. 更完善的模板导出。
3. 更稳定的托管式运行和事件推送。

这三项不阻塞前端第一版，但会决定后续体验上限。

## 建议给员工的开发顺序

1. 先做工作台首页。
2. 再做新建任务向导。
3. 再做任务详情页和历史页。
4. 最后补设置页和导出页。

## 一句话要求

前端不要做成“介绍 CLM 的页面”，要做成“直接能驱动 CLM 干活的控制台”。
