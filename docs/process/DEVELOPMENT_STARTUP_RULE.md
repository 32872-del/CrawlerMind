# CLM 开发前死命令

更新时间：2026-05-20

## 死命令

每次开始任何开发任务之前，必须先阅读这两个文件：

```text
PLAN.md
docs/plans/2026-05-20_AI_MANAGED_CRAWL_LOOP_V2_SHORT_TERM_PLAN.md
```

读完之后，才能决定当前任务怎么做。

## 为什么必须读

CLM 当前最重要的短期目标不是继续零散堆工具，而是完成：

```text
AI Managed Crawl Loop v2
```

也就是把已有采集能力组织成一个 AI 可以全程驾驶的闭环：

```text
证据采样 -> AI 结构化决策 -> 可执行动作 -> 运行结果 -> 质量诊断 -> 修复 patch -> 重跑 -> 导出
```

如果没有先读这两个文件，很容易又偏回“左边补一点、右边补一点”的发散开发，最后能力很多，但产品主流程仍然断。

## 开发前必须回答

每次认领任务前，必须先在自己的任务说明或开发日志里回答：

1. 当前任务属于 `AI Managed Crawl Loop v2` 的哪一步？
2. 它解决的是哪个真实断点？
3. 它会接入哪个现有流程、API、runner、profile 或前端页面？
4. 任务完成后，用户能看到什么闭环改进？
5. 如果不做这个任务，当前主流程会卡在哪里？

回答不清楚，就不要开工，先重新审题。

## 开发时禁止

- 禁止脱离主流程随意新增孤立模块。
- 禁止只写分析报告、不把能力接入运行链路。
- 禁止只给自然语言建议，不产出可执行结构。
- 禁止只修局部按钮或局部字段，却不验证完整工作流。
- 禁止绕开当前主线去做低优先级能力。
- 禁止为了“看起来做了很多”而扩大无关范围。

## 开发时必须

- 必须实事求是判断当前代码已经有什么、缺什么。
- 必须优先打通闭环，而不是继续堆散件。
- 必须让 AI 的输入、决策、动作、结果可以被记录和回放。
- 必须让前端或 API 能看到任务状态和执行过程。
- 必须保留测试、日志或文档证据。
- 必须避免破坏已有入口和已有训练能力。

## 完成标准

一个开发任务不能只算“代码写完”。必须至少满足：

```text
实现 -> 测试 -> 文档/日志 -> 复盘当前闭环位置
```

如果任务涉及用户流程，还必须验证：

```text
入口 -> 执行 -> 状态可见 -> 结果/失败原因 -> 下一步动作
```

如果任务涉及 AI，必须验证：

```text
evidence -> llm decision -> structured action -> execution result
```

## 当前优先闭环

当前所有开发优先服务这条链路：

```text
配置模型
-> 输入站点
-> 分析站点
-> 发现/导入目录
-> 选择字段
-> 试跑
-> 质量诊断
-> AI 修复
-> 重跑
-> 全量运行
-> 导出
```

如果某个任务不能帮助这条链路变得更完整、更稳定、更可见，就默认不是当前优先级。
