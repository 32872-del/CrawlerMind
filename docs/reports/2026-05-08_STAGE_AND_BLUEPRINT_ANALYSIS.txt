CLM 阶段与蓝图目标分析

日期: 2026-05-08

一、最近成果

最近这段时间，项目最重要的变化不是“多写了几段代码”，而是
整个工程开始从脚手架变成一个真正能跑、能验收、能持续迭代的
MVP。

已经完成并稳定落地的成果包括:

- LangGraph 主链路跑通: Planner -> Recon -> Strategy -> Executor ->
  Extractor -> Validator
- 百度热搜前 30 条真实 smoke 测试通过
- FastAPI 背景任务可用，接口能异步提交和查询
- SQLite 结果持久化可用
- 浏览器兜底能力可用
- 可选 LLM Advisor 接入可用
- OpenAI-compatible provider adapter 已接好
- 错误码、测试、日报、看板、交接都已经形成规范
- real-site training ladder 已经正式纳入团队工作流

训练侧最近也很关键:

- JSONPlaceholder、Reddit JSON、Countries GraphQL 证明了公共 JSON /
  GraphQL 路径能走通
- AniList 和 Bilibili 证明了公共 API / GraphQL 在更真实场景下也能
 正常工作
- Douban Top250、React docs、Vue examples 证明静态排名页和 SSR /
  framework recon 也已经进入可用范围

二、当前能力

按现在的状态，这个项目已经不是“只能看设计图”的阶段，而是已经
具备一个可用采集系统的最小闭环。

当前可用能力可以概括为:

1. 静态页面采集
2. 浏览器渲染兜底
3. 公共 JSON / GraphQL 采集
4. 基础 recon 和 selector 推断
5. 结果校验与重试
6. 结果持久化和历史查询
7. FastAPI 任务提交与后台执行
8. 可选 LLM 参与规划与策略建议
9. 团队化协作流程: 任务、验收、日报、交接、记忆

如果按蓝图的层级看:

- Level 1 HTML Pipeline: 已完成
- Level 2 Browser Rendering: 已完成 MVP
- Level 3 Visual Page Understanding: 还没开始
- Level 4 Site Mental Model: 还没开始
- Level 5 Autonomous Agent: 只有早期切片，已经可用，但还不完整

三、目前不足

现在最明显的不足，不是“不能跑”，而是“面对复杂站点时还不够稳”。

主要缺口有:

- 动态页、SPA、virtualized list 的真实训练还不够
- API 逆向和网络观测能力还偏薄
- Cloudflare、登录、验证码等复杂场景还只是诊断级，不是稳定处理级
- 任务状态仍然是本地进程内存，重启会丢
- 站点记忆和失败复用还只是文件化思路，没有自动回收成策略
- 前端配置入口还没做，用户操作仍偏开发者视角
- 自动化协作还靠文档纪律，不靠系统级锁和调度

四、下一步计划

下一步最有价值的不是急着扩大功能名词，而是把现有能力往真实
网站场景里继续压实。

优先级建议如下:

1. 继续训练受控 SPA
2. 训练 virtualized list 和无限滚动
3. 补 browser network observation 骨架
4. 强化 API / GraphQL 发现与分页
5. 把失败样本沉淀成 fixture 和测试
6. 继续完善站点记忆和策略复用
7. 再考虑更完整的前端配置入口

五、未来目标

蓝图真正想要的，不只是一个爬虫，而是一个会“观察、选择、记住、
改进”的采集 Agent。

长期目标可以拆成几层:

- 短期: 让项目稳定可演示、可交接、可继续开发
- 中期: 让它能应对更多网站类型，尤其是动态站点和复杂 API 场景
- 长期: 让系统具备站点记忆、失败复用、自我修复、跨页面规划能力
- 最终: 让它从“可配置爬虫”走到“真正能做判断的采集 Agent”

六、我的判断

如果只看“能不能用”，现在已经能用了。

如果看“能不能当作一个小型产品持续迭代”，现在也成立。

如果看“是不是已经接近完整蓝图”，还没有。
现在更像是:
- 地基已经打完
- 主体骨架已经立起来
- 早期房间已经能住
- 但外墙、智能系统、长期维护体系还在继续施工

七、结论

这个项目已经从“想法”变成了“能工作的系统”。
它还没到最终形态，但已经具备继续往上长的底盘。

下一阶段最重要的，不是追求概念完整，而是继续用真实站点训练、
验证、沉淀，让它逐步从 MVP 走向稳定的采集 Agent。
